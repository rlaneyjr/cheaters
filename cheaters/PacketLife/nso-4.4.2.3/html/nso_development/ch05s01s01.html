Key Features of CDB

By default, NSO stores all configuration data in CDB. There are a number of advantages to CDB compared to using a traditional external storage for configuration data. CDB was built for network device configuration in mind, featuring:

  • A solid model for handling configuration data in network devices.

  • Close integration with NSO with no maintenance needs.

  • Fast lightweight database access by keeping the configuration in RAM.

  • Persistence using journal files on disk.

  • A well defined API for writing instrumentation functions to access data.

  • A subscription mechanism enabling external entities to be notified of changes (see the section called “Subscriptions” for details).

  • Support for handling operational, non configuration data (see the section called “Operational Data in CDB” for details).

  • A networked API making it is possible for an unconfigured device to find the configuration data on the network and use that configuration.

  • Automatic support for upgrade and downgrade of configuration data. This is useful not only when performing actual up/downgrades of NSO itself, but also simplifies the development process by allowing individual developers to add/delete items in the configuration without any impact on other developers (see the section called “Automatic Schema Upgrades and Downgrades” for details).

Important

For reliable storage of the configuration on disk, CDB requires that the file system correctly implements the standard primitives for file synchronization and truncation. For this reason (as well as for performance), NFS or other network file systems are unsuitable for use with CDB - they may be acceptable for development, but using them in production is unsupported and strongly discouraged.

Note

Since CDB is a RAM database, the amount of RAM needed is proportional to the number of devices and services managed. When NSO is used to manage a large network the amount of RAM needed can be quite large.

When using CDB to store the configuration data, the applications need to be able to:

  1. Read configuration data from the database.

  2. React to changes to the database. There are several possible writers to the database, such as the CLI, NETCONF sessions, the Web UI, either of the NSO sync commands, alarms that get written into the alarm table, NETCONF notifications that arrive at NSO or the NETCONF agent.

Figure 5, “NSO CDB Architecture Scenario” illustrates the architecture when CDB is used. The Application components reads configuration data and subscribes to changes to the database using a simple RPC-based API. The API is part of the Java library and is fully documented in the Javadoc for CDB.

Figure 5. NSO CDB Architecture Scenario


While CDB is the default data store for configuration data in NSO, it is possible to use an external database, if needed. See the example examples.ncs/getting-started/developing-with-ncs/6-extern-db for details.
